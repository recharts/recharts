Theming

This guide documents the current Recharts theme system. Use RechartsThemeProvider to apply legacyTheme, lightTheme, darkTheme, or your own RechartsTheme object to a subtree of charts.

The core precedence model is now part of the implementation: explicit component props win over theme defaults, theme defaults win over component defaultProps, and the nearest theme provider wins over outer providers.

This preview uses the current Recharts theme API. Switching themes updates axes, grid, tooltip, legend, and the default series styling together.

Built-in themes

Recharts currently ships with three named themes:

  • legacyTheme preserves the long-standing defaults and remains the implicit baseline when no theme provider is present.
  • lightTheme is the new first-party light theme.
  • darkTheme is the matching dark theme.

Built-in themes are plain objects, so they can be imported, inspected, spread, serialized, and used as the starting point for project-specific variants.

Quick start

The main API is a provider around a group of charts. This keeps dashboards, apps, and documentation sites consistent while keeping all existing chart props available as escape hatches.

For one-off local changes, use a nested provider around the chart or chart section that needs different tokens. That is the supported local-override mechanism today; there is no separate chart-root theme prop.

Theme anatomy

A Recharts theme is a plain object shaped like the exported RechartsTheme type. Shared tokens come first, component-specific overrides come second.

Creating custom themes

Recharts does not currently ship a createRechartsTheme() helper. The supported approach is to compose plain objects in user code, usually by spreading lightTheme or darkTheme and overriding the fields you want to change.

If you only need local overrides, prefer a nested provider over cloning the whole theme. If you need a reusable branded theme package, export a plain RechartsTheme object from your own codebase.

Handling multiple series

Multi-series charts must not depend on React child order or render order for color assignment. Recharts now derives a stable series identity from coalesce(dataKey, name).

  1. Explicit component props still win if you set them.
  2. Otherwise, Recharts checks theme.series.keyed[identity].
  3. If there is no keyed override, Recharts hashes the identity into theme.series.palette.

Different graphical items apply the resolved series color differently: Line uses it as stroke; Bar, Scatter, Pie, Funnel, and RadialBar use it as fill; Area and Radar use it as both fill and stroke unless the keyed style provides separate values.

If two series intentionally share the same dataKey and name, that identity is ambiguous by definition. In that case, use explicit fill or stroke props.

Switching themes at runtime

Runtime switching is a normal React state update: change the provider's theme prop and leave chart markup alone. This is the same pattern now used by the Recharts website itself.

Using Recharts themes without a UI library

Recharts works well in plain React apps with no design-system provider. The two practical options are CSS variables and serializable JSON theme objects.

CSS variables make theme switching instant and align naturally with server-rendered HTML. JSON theme objects make it easy to share branded chart defaults across projects, CDNs, CMS pipelines, or design-token builds.

Integrations with popular UI libraries

Recharts currently ships a first-party helper for Ant Design. For the other libraries, the recommended approach today is to read the active library theme in your app, map its tokens to a plain RechartsTheme object, and pass that object to RechartsThemeProvider.

Material UI

Material UI already has the provider-and-theme-object model that Recharts needs. A small local mapper can keep chart surfaces, typography, and series colors aligned with the rest of the app.

Ant Design

Ant Design exposes seed tokens, derived tokens, and algorithms such as defaultAlgorithm, darkAlgorithm, and compactAlgorithm. For the live preview below, Recharts also vendors adapted copies of all seven custom Ant Design website ThemePreview themes.

Ant Design
Default and dark come from Ant Design's built-in algorithms. Bootstrap, Cartoon, Geek, Glass, Illustration, MUI-like, and shadcn-like are adapted from the Ant Design website ThemePreview sources. The chart itself is styled implicitly by RechartsThemeProvider using rechartsFromAntd(token).

The Bootstrap, Cartoon, Geek, Glass, Illustration, MUI-like, and shadcn-like theme configs used in this preview were adapted from Ant Design's website sources under the MIT license: ant-design/.dumi/pages/index/components/ThemePreview/previewThemes.

Chakra UI

Chakra UI is token-first and CSS-variable-driven, so it works well with a Recharts theme object that references Chakra's semantic variables directly.

Mantine

Mantine themes are deeply merged objects with color scales, radii, shadows, fonts, and component defaults. A small mapper keeps chart defaults consistent with the rest of a Mantine app.

shadcn/ui and Tailwind CSS

Many teams do not use a JavaScript theme provider at all. Instead, they define semantic CSS variables and let Tailwind utilities or shadcn components consume them. Recharts can use that same token layer directly.

Current gaps

The core runtime is in place, but a few convenience layers from the original design brief are still missing:

  • There is no first-party createRechartsTheme() helper yet; theme composition is still normal object spread or an app-specific utility.
  • There are no official adapters for Material UI, Chakra UI, Mantine, or shadcn/Tailwind yet.
  • There is no dedicated chart-root theme prop; local overrides use nested providers.
  • Some reserved tokens are not consumed by core yet, including colors.canvas, radii.legend, radii.surface, shadows.legend, and shadows.surface.
  • custom is useful for app-level coordination, but Recharts does not read it automatically.

Why this design

This implementation balances ergonomics with flexibility:

  • A provider gives apps one place to apply shared defaults.
  • Nested providers keep local overrides simple without introducing a new chart-root prop.
  • Plain theme objects are easy to share, serialize, and version.
  • Explicit chart props still work, so existing public APIs remain the final override layer.
  • Stable series identity avoids the color swaps that index-based palettes cause in real React trees.

Alternatives we considered and rejected

Only a chart-level theme prop

This is easy to explain, but it repeats the same theme object on every chart and makes dashboard-wide switching awkward.

Only a provider with no local override

This works for uniform dashboards, but breaks down when one chart needs a presentation mode, a brand-specific palette, or a localized override inside a mixed page. Nested providers solve that without a second API.

An unstyled mode without shared theme tokens

An unstyled escape hatch is still valuable, but by itself it does not solve the main product problem: consistent shared defaults, simple theme switching, and alignment with existing design systems.

Per-component configuration with no token layer

A giant object full of component-specific defaults becomes hard to read, hard to map from external design systems, and hard to extend. Shared tokens plus targeted component overrides scale better.

Index-only automatic palettes

Coloring series only by position looks nice in demos, but it is unstable when series are reordered, toggled, conditionally rendered, or re-traversed by React during updates. Stable series identity is the safer fallback.
